# 橋本競馬AI

橋本競馬AIは、競馬場別の現場データと、AI研究・買い目・検証・人物補正データを分けて管理する競馬予想データベースです。

AI指数ランキング、今日のレース一覧、実データ入力フォーム、結果検証フォーム、最新結果検証ログ一覧、神レースTOP10、AI指数自動入力、危険人気馬自動判定、神穴ランキング（爆穴候補）、三連単自動生成、WIN5自動生成、回収率ダッシュボード、競馬場別AI、騎手ランキングを横断的に扱い、予想根拠と検証結果を再利用できる構造にしています。

#### Phase2-2 競馬場別・距離別補正テーブル

AI指数自動計算エンジンは、基本点に対して `src/dashboard.js` の4種類の補正テーブルを加算して最終指数を算出します。

- `COURSE_CORRECTION_TABLE`: 中山・東京・京都・阪神・中京・新潟・福島・小倉の主要8場に対し、前有利補正、差し有利補正、内枠補正、外枠補正、坂適性、小回り適性、直線長さを保持します。
- `DISTANCE_CORRECTION_TABLE`: 芝短距離、芝マイル、芝中距離、芝長距離、ダート短距離、ダート1400、ダート1800、ダート中長距離を保持し、芝/ダートと距離から自動カテゴリ判定します。
- `SURFACE_CORRECTION_TABLE`: 芝・ダート・障害の馬場種別ごとの前後バイアス、枠、スピード/スタミナ、危険度補正を保持します。
- `PACE_STYLE_CORRECTION_TABLE`: 逃げ、先行、好位、中団、差し、追込（既存入力の自在は好位相当でも扱う）に対し、AI指数・神穴指数・危険人気馬指数それぞれの脚質補正と、競馬場/距離バイアスへの重みを保持します。

自動計算時は、各馬ごとに「基本点」「競馬場補正」「距離補正」「脚質補正」「最終指数」を `scoreBreakdown` に保存し、出走馬テーブルと各ランキングに補正内訳を表示します。手入力指数がある場合は手入力値を最終指数として優先しつつ、比較用の補正内訳も保持します。

Phase2-3のGitHub JSON保存本実装設計は [`GITHUB_JSON_STORAGE_DESIGN.md`](GITHUB_JSON_STORAGE_DESIGN.md) に整理しています。localStorage中心の保存から、`data/*.json`、GitHub API読込/保存、差分更新、バックアップ、保存方式切替へ段階移行する流れを定義しています。

Phase2-4の自己進化AI強化設計は [`SELF_EVOLUTION_AI_DESIGN.md`](SELF_EVOLUTION_AI_DESIGN.md) に整理しています。バックテスト、結果検証、OSアップデートを使って、AI指数や各AI補正、危険人気馬、神穴、WIN5、競馬場OS、距離別OSを継続的に改善する流れを定義しています。

Phase2-5のiPad実戦運用マニュアルは [`IPAD_OPERATION_MANUAL.md`](IPAD_OPERATION_MANUAL.md) に整理しています。iPadだけで週末の出馬表確認、CSV取込、AI指数確認、神穴確認、危険人気馬確認、三連単/WIN5生成、結果検証、自己進化ログ保存まで行う流れを定義しています。

CSV取込強化、netkeiba手動CSV変換、JRA出馬表データ取込準備、オッズ取込、AI指数/神穴指数/危険人気馬の自動計算、三連単/WIN5自動生成精度向上、`data/*.json` 保存、GitHub API保存、自己進化AI強化、iPad週末運用フローを段階的に進めます。

### Phase2-1 CSVバリデーション仕様

実データ入力フォームの「CSV取込バリデーション」では、CSVを選択した時点で取込前検証を行い、プレビューに各行の `OK` / `注意` / `エラー` を表示します。

**必須カラム**

- `開催日`
- `競馬場`
- `レース番号`
- `レース名`
- `馬番`
- `馬名`
- `騎手`
- `人気`
- `オッズ`
- `脚質`
- `距離`
- `芝ダート`
- `馬場状態`

必須カラムが不足している場合、赤色のエラー警告を表示し、CSV取込ボタンを無効化します。必須カラムが存在していても、各行で必須値が空欄の場合は行エラーとして扱い、取込を停止します。

**任意カラム**

- `調教師`
- `調教評価`
- `AI指数`
- `神穴指数`
- `危険人気馬指数`
- `想定4角位置`

任意カラムがない場合、黄色の注意表示を出しますが取込は可能です。任意カラムが存在して空欄の行も注意としてプレビューに表示し、既存フォームの初期値・補完値で取り込みます。

**数値チェック**

以下のカラムは数値形式を検証します。

- `馬番`
- `人気`
- `オッズ`
- `AI指数`
- `神穴指数`
- `危険人気馬指数`

`馬番`、`人気`、`オッズ` は必須のため、空欄または数値以外の場合はエラーです。`AI指数`、`神穴指数`、`危険人気馬指数` は任意カラムですが、カラムが存在して値が入っている場合は数値以外をエラーとして扱います。

**取込後の反映**

エラーがないCSVを取り込むと、先頭行の `開催日`、`競馬場`、`レース番号`、`レース名`、`距離`、`芝ダート`、`馬場状態` をレース登録フォームへ反映し、各行の出走馬を入力済み出走馬テーブル、AI指数ランキング、神穴指数ランキング、危険人気馬ランキング、三連単生成に連動させます。黒×金デザインとスマホ表示に合わせ、警告・注意・プレビューもレスポンシブなカード/テーブル表示にしています。


## GitHub Pages公開手順

このリポジトリは、`main` ブランチ直下の `index.html` をGitHub Pagesの公開対象にする前提です。`docs/` フォルダを作成したり、公開対象に切り替えたりする必要はありません。

### 公開構成

GitHub Pagesでは、以下のファイルをリポジトリ直下から相対パスで読み込みます。

- `index.html`
- `manifest.json`
- `icon.svg`（仮アイコン。本番用のアイコン画像は後で追加）
- `src/dashboard.css`
- `src/dashboard.js`
- `src/storage/localStorageAdapter.js`
- `src/storage/githubAdapter.js`
- `src/storage/dataStorageService.js`
- `data/*.json`

`index.html` からCSS/JSは `./src/...`、自己進化ログJSONは `./data/selfEvolutionLogs.json` のように相対パスで参照するため、GitHub Pagesのリポジトリ配下URLでもPC、自宅PC、iPad、スマホから同じURLで開けます。

### GitHub Pages設定

1. GitHubの対象リポジトリを開く
2. **Settings** を開く
3. **Pages** を開く
4. **Build and deployment** を確認する
5. **Source: Deploy from a branch** を選択する
6. **Branch: main** を選択する
7. **Folder: /root** を選択する
8. **Save** を押す

### 公開後の確認項目

- トップ画面が開く
- 黒×金デザインが反映される
- AI指数ランキングが見える
- CSV取込画面が見える
- iPadで開ける
- ホーム画面に追加したアイコンから、アプリ風に単独起動できる

### Pages公開チェック画面での確認手順

GitHub Pagesを公開したら、トップ画面の **Pages確認** または実運用モード操作メニューの **Pages公開チェック** から `#pages-publish-check` を開きます。

1. **GitHub Pages公開URL** に公開URL（例: `https://ユーザー名.github.io/hashimoto-keiba-ai/`）を入力する
2. **公開チェックを実行** を押す
3. 以下の自動チェックが `OK` / `注意` / `未確認` のどれで表示されるか確認する
   - `index.html` 読み込み
   - CSS読み込み
   - JavaScript読み込み
   - `manifest.json` 読み込み
   - `data/selfEvolutionLogs.json` などのdata JSON読み込み
   - localStorage利用可
   - iPad表示対応
   - スマホ表示対応
4. `注意` が出た項目は、公開URL、GitHub Pagesの反映待ち、相対パス、ブラウザのlocalStorage設定を確認する
5. iPadでは同じ画面の **iPad確認メモ** で次を手動チェックする
   - Safariで開いたか
   - ホーム画面に追加したか
   - 黒×金デザインが表示されたか
   - CSV取込画面が見えるか

`Pages公開チェック` は既存の黒×金デザインとレスポンシブCSSの中に追加しているため、PC・iPad・スマホの表示確認画面としてそのまま利用できます。

### iPad・iPhoneでホーム画面に追加する手順

GitHub Pages公開後は、Safariからホーム画面へ追加すると、`manifest.json` とiOS向けmeta設定によりアプリ風に起動できます。

1. SafariでGitHub PagesのURLを開く
2. 共有ボタンを押す
3. ホーム画面に追加
4. 名前を「橋本競馬AI」にする
5. ホーム画面から起動

現在のアイコンは `icon.svg` の仮SVGアイコンです。黒×金デザインに合わせた本番用のアイコン画像は後で追加してください。

## ダッシュボード

トップ画面は `index.html` から開けます。黒×金の競馬AI管理画面として、スマホ表示でも主要判定を確認できるレスポンシブ構成です。


### iPad・スマホでの使い方

- **iPad横画面**: 画面幅がタブレット帯になると、専用クイックUIが2カラムで表示されます。左カラムに「今日のレース」を固定し、右カラムに「AI指数」「神穴」「危険人気馬」「買い目」を並べるため、レースを選びながら指数・穴馬・危険馬・買い目を同時に確認できます。
- **スマホ縦画面**: 専用クイックUIは縦スクロール中心の1カラムに切り替わり、重要カードを「今日のレース」「AI指数」「神穴」「危険人気馬」「買い目」の順に確認できます。ボタンと入力フォームはタップしやすい高さに拡大しています。
- **下部固定ナビゲーション**: iPad・スマホでは画面下部に「ホーム」「今日のレース」「AI指数」「買い目」「保存ログ」の固定メニューが出ます。長い画面でも主要セクションへすぐ戻れます。
- **重要操作の固定表示**: 下部ナビの上に「保存」「買い目生成」「結果検証」「バックアップ」の固定ボタンを表示します。レース入力中や検証中でも、スクロール位置に関係なく重要操作へ移動できます。
- **PC / iPad / スマホの切り替え**: PCは従来の広いダッシュボードを優先し、iPadは2カラム、スマホは1カラムへCSS media queryで切り替えます。既存の黒×金デザインとカード構成は維持しています。

- **実運用モード操作メニュー**: トップ画面から「今日のレース一覧」「実データ入力フォーム」「神レースTOP10」「AI指数自動入力」「危険人気馬自動判定」「神穴ランキング」「三連単自動生成」「WIN5自動生成」「回収率ダッシュボード」「結果検証フォーム」「最新結果検証ログ一覧」「AI精度テスト・バックテスト」「OS自動アップデート管理」「バックアップ／復元」へ直接移動できます。
- **今日のレース一覧**: 開催場、発走時刻、距離、AI勝負度、注目馬、推奨操作、状態を一覧化し、指数入力・神穴確認・三連単生成・WIN5生成へつなげます。
- **実データ入力フォーム**: 開催日、競馬場、レース番号、レース名、距離、芝/ダート、馬場状態、天候、頭数を登録し、馬番、馬名、騎手、人気、オッズ、脚質、想定4角位置、調教評価、AI指数、神穴指数、危険人気馬指数を出走馬ごとに入力できます。
- **自動ランキング / 買い目生成**: 入力された出走馬をAI指数順、神穴指数順、危険人気馬指数順で即時ランキング表示し、三連単、WIN5、神穴候補、危険人気馬をブラウザ上で自動生成します。
- **localStorage保存**: 保存ボタンで `hashimoto-keiba-ai:real-race-entry:v1` にレース・出走馬・生成時刻を保存します。保存処理はStorage Adapterとして分離しているため、後からGitHub API保存へ拡張できます。
- **GitHub保存設定（準備実装）**: ダッシュボードの「GitHub保存設定」で保存方式を `localStorage` / `GitHub` から選択し、GitHubユーザー名、リポジトリ名、ブランチ名、保存先ディレクトリ、アクセストークンを登録できます。GitHub選択時は `src/storage/githubAdapter.js` 経由でGitHub Contents API形式の読み書きを呼び出します。
- **結果検証フォーム**: 1〜3着の馬番・馬名・騎手・人気、距離、芝/ダート、三連単配当、馬場状態、実ペース、通過順を入力し、本命・対抗・穴馬・神穴・危険人気馬・三連単・WIN5の事前予想照合をチェックできます。
- **OSアップデート欄**: 今回の学習点、次回から採用するルール、保留ルール、削除するルールを検証フォーム内で記録し、競馬場OSやAI研究所へ反映する材料として残します。
- **結果検証ログ自動保存**: 結果検証フォームの入力は `hashimoto-keiba-ai:result-verification-logs:v1` にResultLogStorage Adapter経由で自動保存します。JSONは `race`、`result`、`predictionCheck`、`osUpdate` に分離しているため、GitHub API保存へ拡張しやすい構造です。
- **最新結果検証ログ一覧**: localStorageに保存された検証ログを競馬場別・日付別にグループ表示し、競馬場と日付のフィルターで最新ログを絞り込めます。
- **AI精度テスト・バックテスト**: localStorageの結果検証ログと回収率ログを対象に、期間、競馬場、距離、芝/ダート、馬場状態、券種で絞り込み、本命・対抗・神穴・危険人気馬・三連単・WIN5の的中率、総投資額、総払戻額、ROI、券種別ROI、競馬場別ROI、弱点分析、AI補正案を集計します。
- **OS自動アップデート管理画面**: 競馬場、距離、芝/ダート、馬場状態、ペース、勝ち位置ゾーン、有利脚質、危険人気馬条件、万馬券パターン、採用/保留/削除、根拠レース、ルール本文を登録し、競馬場別・距離別・馬場別・ステータス別にOSルール一覧を確認できます。
- **結果検証からOSアップデートへ連動**: 結果検証フォームの「今回の学習点」「次回から採用するルール」「保留ルール」「削除するルール」は、「OS管理へ反映」ボタンでOSルールへ変換します。採用ルールは採用、保留ルールは保留、削除するルールは削除として保存し、学習点のみの場合は保留仮説として登録します。
- **OSルールlocalStorage保存**: OSルールは `hashimoto-keiba-ai:os-update-rules:v1` にOSRuleStorage Adapter経由で保存します。JSONはルール本文、条件、ステータス、根拠レース、source情報を分けて持つため、GitHub API保存へ拡張しやすい構造です。
- **バックアップ／復元**: localStorage内のレース登録・出走馬・事前予想ログ・結果検証ログ・OSアップデートログ・WIN5買い目・回収率・自動買い目をJSONでエクスポート／インポートできます。全データ、競馬場別、WIN5のみ、回収率のみの4種類のバックアップボタンを用意し、復元時は上書き前に確認メッセージを表示します。バックアップ履歴には作成日時、保存対象、件数、バックアップ種別が残り、履歴から任意のバックアップをワンクリック復元できます。
- **保存前／復元前バックアップ確認**: GitHub保存前は現在のlocalStorageデータを自動バックアップし、JSONインポートやGitHubデータ読込前にも一時バックアップを作成します。バックアップ履歴の削除時は確認ダイアログを表示し、誤削除を防ぎます。

## データ保存サービス層

現在のダッシュボード保存は `src/storage/dataStorageService.js` を入口にし、共通関数 `saveData(type, data)`、`loadData(type)`、`exportData(type)`、`importData(type, data)` で扱います。実体の保存先はアダプターで切り替える設計です。

- **現在の保存先**: `src/storage/localStorageAdapter.js` がブラウザの `localStorage` へJSON保存します。既存の保存キーは維持しているため、これまで保存したレース、買い目、結果検証、OSルール、回収率データを壊さず読み込めます。
- **将来のGitHub保存**: `src/storage/githubAdapter.js` は未接続の雛形です。将来、GitHub Contents APIへ接続し、`/data/*.json` を読み書きする実装を追加します。未接続状態では保存・読込・エクスポート・インポートをエラーにして、誤送信を防ぎます。
- **保存先マッピング確認**: `HashimotoDataStorage.getRepositoryFileMap()` で `raceEntries` → `/data/raceEntries.json` のようなタイプ別保存先を取得できるため、GitHub連携実装時も対象ファイルを共通定義から参照できます。
- **アダプター切り替え**: `HashimotoDataStorage.setActiveAdapter("localStorage")` / `HashimotoDataStorage.setActiveAdapter("github")` の形で保存先を切り替えられる構造にしています。GitHub側は現在エラーを返し、誤って未実装APIへ送信しないようにしています。
- **対象タイプ**: `raceEntries`、`horseEntries`、`predictions`、`results`、`raceResults`、`osUpdates`、`win5Tickets`、`roiRecords`、`betTickets`、`backupData`、`selfEvolutionLogs` を共通タイプとして定義しています。

## バックアップ／復元機能の使い方

1. ダッシュボードの「バックアップ」または「バックアップ／復元」へ移動します。黒×金デザインの操作パネルで、スマホではバックアップ欄と復元欄が1カラム表示になります。
2. **全データをバックアップ**: `localStorage` に保存済みのレース登録データ、出走馬データ、事前予想ログ、結果検証ログ、結果CSV取込データ、OSアップデートログ、WIN5買い目、回収率データ、自動買い目データをまとめてJSONダウンロードします。
3. **競馬場別バックアップ**: 競馬場セレクトで東京・中山などを選び、その競馬場に一致するレース登録、検証ログ、OSルール、回収率、WIN5内レース、自動買い目を抽出してJSON化します。競馬場名は「東京」と「東京競馬場」のどちらの形式でも照合できるようにしています。
4. **WIN5のみバックアップ**: `hashimoto-keiba-ai:win5-ticket:v1` のWIN5対象5レース、候補ゾーン、採用フォーメーション、点数、推奨資金をJSONで保存します。
5. **回収率のみバックアップ**: `hashimoto-keiba-ai:investment-results:v1` の投資結果、払戻、ROI、損益、的中率集計に使うデータをJSONで保存します。
6. **復元**: 「JSONファイルを選択」でバックアップJSONを選び、「読み込んだJSONから復元」を押します。復元前に確認メッセージが出るため、内容を確認してから実行してください。実行後は対象のlocalStorageキーが復元され、レース入力、WIN5、三連単、回収率、結果検証ログ、OSルール画面を再描画します。
7. 復元は同名キーを上書きします。誤復元を避けるため、復元前には必ず「全データをバックアップ」で現状JSONを保存してください。
8. 将来GitHubへJSON保存する場合は、`backupStorageAdapters.github.save` と `backupStorageAdapters.github.restore` をGitHub APIに接続します。バックアップpayloadには `repositoryPath`、`provider`、`extensibleProvider`、`keys`、`data` を分けて持たせているため、localStorage保存からGitHub保存へ移行しやすい構造です。

## 回収率管理と資金配分AIの使い方

1. ダッシュボードの「回収率」または「回収率・資金管理ダッシュボード」へ移動します。黒×金デザインのまま、スマホでは入力フォーム、サマリー、各集計テーブルが1カラムへ切り替わります。
2. 「投資結果入力」フォームに、日付、競馬場、レース番号、券種、購入点数、投資金額、払戻金額、的中/不的中、メモを入力して「結果を保存」を押します。
3. 保存後、ROI、利益/損失、累計投資額、累計払戻額、累計利益、的中率が即時に自動計算されます。ROIは `払戻金額 ÷ 投資金額 × 100`、利益/損失は `払戻金額 - 投資金額` で表示します。
4. 券種別集計は、三連単、WIN5、馬連、三連複、単勝、複勝の6券種を固定表示します。投資がない券種も0円・0%として残すため、買っていない券種と改善対象を見分けやすくしています。
5. 競馬場別集計は、中山、東京、京都、阪神、中京、新潟、福島、小倉の主要8場を固定表示します。各場ごとの投資額、払戻額、ROI、的中率を比較し、競馬場OSの改善材料にします。
6. 月別ROIでは、月別投資額、月別払戻額、月別回収率、月別利益を表示し、棒グラフで月ごとの回収率推移を確認できます。
7. 資金配分AIは、累計ROI、平均投資額、累計利益をもとに「勝負レース」「回収レース」「見送りレース」の推奨投資額を表示します。勝負レースは期待値最大のレースに集中、回収レースは標準配分、見送りレースは0円で資金温存する運用です。
8. 入力データは `hashimoto-keiba-ai:investment-results:v1` にlocalStorage保存します。保存JSONは `results`、`computed`、`repositoryPath`、`provider` を分けたGitHub保存拡張対応構造です。将来GitHubへ保存する場合は、`investmentStorageAdapters.github` の `save/load` をGitHub APIに接続します。
9. 「入力クリア」はフォームだけを初期化します。「保存データ削除」はブラウザ内の投資結果保存ログを削除するため、正式な運用ログへ移管済みか確認してから実行してください。


## 三連単自動買い目生成エンジンの使い方

1. ダッシュボードの「実データ入力フォーム」で開催日、競馬場、レース番号、レース名、距離、馬場状態、頭数を登録します。
2. 出走馬ごとに馬番、馬名、騎手、人気、オッズ、脚質、想定4角位置、調教評価、AI指数、神穴指数、危険人気馬指数を入力します。3頭以上入力すると三連単エンジンが即時起動します。
3. 1着候補はAI指数上位をAゾーン、次位をBゾーン、神穴・人気薄をCゾーンとして自動抽出します。危険人気馬指数80以上かつ5人気以内の馬は、危険人気馬除外ルールにより1着候補から外します。
4. 2着候補はAI指数上位から抽出しますが、1人気は原則2着固定しません。1人気は1着候補または3着候補として扱い、2着候補リストから自動除外します。
5. 3着候補はAI指数上位、神穴指数上位、1人気を合成して作ります。神穴指数が高い馬や人気薄・高オッズ馬は「神穴3着」として万馬券型へ優先配置します。
6. 点数制御は頭数で自動切り替えます。10頭以下は8点、11〜14頭は12点、15〜18頭は16点を基本にし、100円単位の推奨資金を表示します。
7. 買い目一覧は「本線」「攻撃型」「万馬券型」に分けて表示します。本線はAゾーン頭、攻撃型はB/Cゾーン頭、万馬券型はB/Cゾーン頭×神穴3着を優先します。
8. 「三連単買い目保存」を押すと `hashimoto-keiba-ai:trifecta-tickets:v1` にlocalStorage保存します。保存JSONは `race`、`candidates`、`tickets`、`summary`、`repositoryPath`、`provider` を分けたGitHub保存拡張対応構造です。将来GitHubへ保存する場合は、`trifectaStorageAdapters.github` の `save/load` をGitHub APIに接続します。
9. 黒×金デザインを維持し、スマホでは候補一覧と本線・攻撃型・万馬券型の買い目一覧が1カラム表示へ切り替わります。

## WIN5自動生成システムの使い方

1. ダッシュボードの「WIN5自動生成」へ移動し、レース1〜レース5それぞれに競馬場、レース番号、レース名、距離、馬場、頭数を入力します。
2. 各レースの候補馬欄へ、A固定候補、B本線候補、Cゾーン候補、D爆発候補、危険人気馬を改行またはカンマ区切りで入力します。馬番だけでも、馬番＋馬名でも運用できます。
3. 生成ルールは「A固定2・B本線1・C狙い1・D警戒1」です。各レースでAを最大2頭、B/C/Dを各最大1頭採用し、重複候補は1頭として点数計算します。
4. 入力するとWIN1〜WIN5のフォーメーション、各レースの採用頭数、WIN5点数、100円単位の推奨資金が自動更新されます。5レースすべてに採用候補がある場合のみ点数を確定します。
5. 配当ラインは、採用ゾーンと点数から「100万ライン」「500万ライン」「3000万ライン」「1億ライン」を自動判定します。D爆発候補を含む少点数構成ほど高配当ラインとして表示します。
6. 「買い目保存」を押すと `hashimoto-keiba-ai:win5-ticket:v1` にlocalStorage保存します。保存JSONは `races`、`formation`、`summary`、`repositoryPath` を分けたGitHub保存拡張対応構造です。将来GitHubへ保存する場合は、`win5StorageAdapters.github` の `save/load` をGitHub APIに接続します。
7. 黒×金デザインのまま、スマホではレース基本情報、候補馬入力、生成サマリーが1カラムへ切り替わるため、現場入力でも利用できます。

## 結果CSV取込（Phase2-1）

Phase2-1では、既存の出馬表CSV取込とは別に、レース後の実結果を取り込む「結果CSV取込」セクションをダッシュボードへ追加しています。目的は、着順・払戻・4角位置・上がり3F・馬場状態・実ペース・メモを正規化して、結果検証フォーム、バックテスト、AI自己進化ログで同じデータを再利用できるようにすることです。

### 使い方

1. `index.html` の「レース登録・出走馬入力フォーム」内にある「結果CSV取込」へ移動します。既存の「CSV取込バリデーション」は出馬表用で、結果CSVは別セクションとして扱います。
2. 「結果CSVテンプレート」ボタンでテンプレートをダウンロードし、必要カラムを入力します。
3. 結果CSVファイルを選択すると、取り込み前にプレビュー表へ行ごとの `OK` / `注意` / `エラー` が表示されます。
4. 必須カラム不足、必須値の空欄、数値カラムの型エラーがある場合は取り込みボタンを無効化します。任意カラム不足や任意値の空欄は「注意」として表示し、取り込みは可能です。
5. 「結果CSVを取り込む」を押すと、正規化payloadを `raceResults` に保存します。payloadの `type` は `raceResults`、GitHub保存拡張時の想定ファイルは `data/raceResults.json` です。
6. 取り込んだ各レースは結果検証ログ互換の構造へ変換され、`hashimoto-keiba-ai:result-verification-logs:v1` にも連動します。先頭レースは結果検証フォームへ自動反映され、バックテスト集計とAI自己進化ログの材料にもなります。

### カラム定義

| カラム | 必須 | 型/形式 | 用途 |
| --- | --- | --- | --- |
| 開催日 | 必須 | `YYYY-MM-DD`推奨 | レース識別、期間フィルター、バックテスト対象選択 |
| 競馬場 | 必須 | 例: `東京競馬場` / `東京` | 競馬場別検証、ROI弱点分析、自己進化ログ根拠 |
| レース番号 | 必須 | 数値 | レース識別、結果検証フォーム反映 |
| レース名 | 任意 | 文字列 | 結果検証ログの表示名。未入力時は「結果CSV取込レース」で補完 |
| 着順 | 必須 | 数値 | 1〜3着を結果検証フォームの着順欄へ反映 |
| 馬番 | 必須 | 数値 | 着順馬の馬番 |
| 馬名 | 必須 | 文字列 | 着順馬名 |
| 騎手 | 任意 | 文字列 | 1〜3着の騎手欄、検証ログ詳細 |
| 人気 | 任意 | 数値 | 人気帯別の弱点分析、結果フォームの人気欄 |
| オッズ | 任意 | 数値 | 高配当・妙味検証用の生データ |
| 4角位置 | 任意 | 数値 | 通過順サマリー、展開/位置取りの検証 |
| 上がり3F | 任意 | 数値 | 末脚・上がり性能の検証 |
| 三連単配当 | 任意 | 数値 | 払戻検証、ROI/三連単検証の補助値 |
| 馬場状態 | 任意 | `良` / `稍重` / `重` / `不良` など | 馬場別バックテスト、OSルール反映 |
| 実ペース | 任意 | `超スロー` / `スロー` / `平均` / `ハイ` / `超ハイ` など | ペース別弱点分析、自己進化ログ |
| メモ | 任意 | 文字列 | 結果検証フォームの学習点、自己進化ログ補助メモ |

### 保存構造

`raceResults` payloadは、CSV行単位の `items` と、開催日・競馬場・レース番号・レース名で束ねた `races` を持ちます。`races` から1〜3着を抽出して結果検証ログ互換データを作るため、バックテストや自己進化ログの読み込み口を壊さずに拡張できます。

```json
{
  "storageVersion": 1,
  "type": "raceResults",
  "provider": "localStorage",
  "storageKey": "raceResults",
  "items": [],
  "races": []
}
```

## AI精度テスト・バックテスト機能の使い方

1. `index.html` を開き、トップメニューから「AI精度テスト・バックテスト」へ移動します。黒×金デザインのカードとして表示され、スマホでは対象選択、集計表、弱点分析、改善提案が1カラムに切り替わります。
2. 先に「結果検証フォーム」で検証日、競馬場、距離、芝/ダート、馬場状態、実ペース、1〜3着人気、事前予想照合（本命・対抗・神穴・危険人気馬・三連単・WIN5）を保存します。これが的中率・弱点分析の元データになります。
3. 「回収率ダッシュボード」で投資結果を保存します。券種、競馬場、投資額、払戻額、的中/不的中を保存すると、総投資額、総払戻額、ROI、利益/損失、券種別ROI、競馬場別ROIに反映されます。
4. バックテスト対象選択で、期間、競馬場、距離帯（短距離・マイル・中距離・長距離）、芝/ダート、馬場状態、券種を選び、「バックテスト集計」を押します。フォーム入力中も即時再集計されます。
5. 的中率集計では、本命的中率、対抗的中率、神穴的中率、危険人気馬判定成功率、三連単的中率、WIN5的中率を表示します。
6. 弱点分析では、苦手競馬場、苦手距離、苦手馬場、苦手ペース、外れやすい人気ゾーンを抽出します。改善提案では、AI指数補正、展開AI補正、危険人気馬AI補正、神穴AI補正、WIN5補正の候補を表示します。
7. 現在の集計対象は `hashimoto-keiba-ai:result-verification-logs:v1`、結果CSV取込の `raceResults`、`hashimoto-keiba-ai:investment-results:v1` のlocalStorage保存ログです。実装は `backtestSourceAdapters.local/github` として読み込み口を分けているため、将来 `data/results.json` や `data/roiRecords.json` のGitHub JSON保存データへ切り替えやすい構造です。

## GitHub API保存設定（準備実装）

将来的にブラウザのlocalStorageではなく、GitHubリポジトリ内の `data/*.json` へ直接保存するための準備実装を追加しています。現在の既定値は従来通り `localStorage` です。

### 設定手順

1. `index.html` を開き、画面上部または操作メニューから「GitHub保存設定」へ移動します。
2. 保存方式で `localStorage` または `GitHub` を選択します。通常運用や未設定時は `localStorage` を選んでください。
3. GitHub保存を試す場合は、以下を入力して「設定を保存」を押します。
   - GitHubユーザー名
   - リポジトリ名
   - ブランチ名（例: `main`）
   - 保存先ディレクトリ（例: `data`）
   - アクセストークン
4. 「接続テスト」ボタンでGitHub API疎通を確認します。設定不足やfetch非対応環境ではモック/準備状態のメッセージを返し、ブラウザでは `https://api.github.com/repos/{owner}/{repo}` へ接続する形にしています。
5. GitHub保存方式を選ぶと、レース登録、出走馬、三連単、WIN5などの保存処理がStorage Adapter経由でGitHub用Adapterを呼び出します。保存先pathは `data/raceEntries.json`、`data/horseEntries.json`、`data/betTickets.json` などの `data/*.json` を想定しています。

### 実装メモ

- GitHub連携関数は `src/storage/githubAdapter.js` に集約しています。`getGithubConfig()`、`setGithubConfig(config)`、`clearGithubConfig()`、`testGithubConnection()`、`readGithubJson(path)`、`writeGithubJson(path, data)`、`createGithubCommit(path, content, message)` を用意しています。
- `src/storage/dataStorageService.js` は現在の保存方式を保持し、`localStorage` の場合は従来Adapter、`github` の場合はGitHub Adapterへ委譲します。
- GitHubへの書き込みはContents APIの `PUT /repos/{owner}/{repo}/contents/{path}` を想定し、既存ファイル更新時に必要な `sha` を事前取得します。

### アクセストークンの注意点

アクセストークンは現段階ではブラウザの `localStorage` に保存します。これは実装準備としての簡易設計であり、共有端末・本番運用・第三者が触れる環境では漏えいリスクがあります。

- 必要最小限の権限を持つfine-grained token、またはGitHub App tokenを使用してください。
- 共有端末では利用後に「設定クリア」を押してください。
- 本番運用ではサーバー側プロキシ、GitHub App、短命トークンなど、トークンをブラウザに長期保存しない構成を推奨します。


## バックアップ確認機能と安全モードの使い方

1. **GitHub保存前の自動バックアップ**: 保存方式を `GitHub` にしてレース登録、三連単買い目、WIN5買い目を保存すると、保存前に現在のlocalStorage全体が「保存前バックアップ」として履歴へ追加されます。安全モードの「GitHub保存前に自動バックアップする」をOFFにすると、この自動作成だけを停止できます。
2. **復元前の一時バックアップ**: JSONインポート、GitHubデータ読込、バックアップ履歴からのワンクリック復元を実行する前に、現在データを「復元前バックアップ」として履歴へ保存します。上書き後に戻したい場合は、直前に作成された履歴を選んで復元してください。
3. **バックアップ履歴画面**: 「バックアップ／復元」カード下部の履歴テーブルで、作成日時、保存対象、件数、バックアップ種別を確認できます。手動バックアップ、保存前バックアップ、復元前バックアップは同じ履歴に最大30件まで保持します。
4. **ワンクリック復元**: 履歴テーブルの「復元」ボタンを押すと、そのバックアップに含まれるlocalStorageキーだけを復元します。復元後は保存方式をlocalStorageへ戻し、ダッシュボード表示を再読み込みします。
5. **削除確認**: 履歴テーブルの「削除」ボタンを押すと確認ダイアログが表示されます。誤って削除しないよう、OKを押した時だけ履歴から削除します。
6. **安全モード設定**: 「GitHub保存設定」内のチェックボックスで、保存前確認、復元前確認、GitHub保存前自動バックアップを切り替えます。共有端末や本番運用では3項目をONのまま使い、検証作業で確認ダイアログを減らしたい時だけ必要に応じてOFFにしてください。

## OSアップデート運用ルール

1. 結果検証後は、必ず「今回の学習点」「次回から採用するルール」「保留ルール」「削除するルール」を分けて入力します。採用・保留・削除を混在させず、1行1ルールを基本にします。
2. OS管理へ反映する時は、結果検証フォームの「OS管理へ反映」ボタンを使います。採用ルール、保留ルール、削除ルールはそれぞれOSルールのステータスへ自動変換し、根拠レースには検証日・競馬場・R番号・レース名・ResultLogIDを残します。
3. 手動登録するOSルールは、競馬場、距離、芝/ダート、馬場状態、ペースを必ず設定します。距離や芝/ダートが結果検証だけでは判別できない場合は、OS管理画面で後から補完します。
4. 採用ルールは次回予想からAI指数補正、危険人気馬判定、万馬券パターン、買い目生成へ反映する候補として扱います。保留ルールは最低2〜3レースの追加検証後に採用または削除へ変更します。削除ルールは旧ロジックの再利用を避けるため、根拠レースと失敗理由を残します。
5. localStorage保存中はブラウザ内の運用メモとして扱い、正式保存時は `各競馬場フォルダ/2026/OSアップデート/` または `競馬場OS/` へMarkdown化して移管します。将来GitHub保存へ拡張する場合は、`osRuleStorageAdapters.github` を実装し、同じJSON構造をGitHub APIへ送信します。

| `騎手データベース/` | 騎手補正に使う成績・適性を管理するDB | 競馬場適性、距離適性、脚質別成績、人気別信頼度、乗り替わり傾向 |
| `調教師データベース/` | 厩舎補正に使う仕上げ・ローテ傾向を管理するDB | 仕上げ傾向、休み明け、叩き2走目、距離変更、競馬場別成績、騎手起用 |

## AI自己進化ループの使い方

ダッシュボードの `AI自己進化ループ` セクションでは、結果検証やバックテストで見つけた改善案を、次回予想へ反映するためのログとして保存・確認できます。

1. `index.html` をブラウザで開き、上部メニューまたは運用メニューから `AI自己進化ループ` へ移動します。
2. 既定データは `data/selfEvolutionLogs.json` から読み込まれます。すでにブラウザへ保存済みのログがある場合は、同じ構造の `localStorage` データを優先表示します。
3. `自己進化ログ追加保存` フォームへ、日付、競馬場、距離、改善対象AI、改善内容、採用/保留/却下、根拠レース、次回反映メモを入力します。
4. `自己進化ログをlocalStorageへ保存` を押すと、`hashimoto-keiba-ai:self-evolution-logs:v1` に追記保存され、一覧へ即時反映されます。
5. 採用ログは次回のAI指数・展開AI・神穴AI・危険人気馬AI・WIN5 AIなどの補正候補として扱い、保留ログは追加レースで再検証、却下ログは同じ誤学習を避ける判断材料として残します。

自己進化ログの必須項目は以下です。

| 項目 | 内容 |
| --- | --- |
| 日付 | 改善判断を登録した日付 |
| 競馬場 | 根拠レースまたは反映対象の競馬場 |
| 距離 | 芝/ダートと距離条件 |
| 改善対象AI | AI指数エンジン、展開AI、神穴AI、危険人気馬AI、WIN5 AIなど |
| 改善内容 | どの条件で何を補正するか |
| 採用状態 | 採用、保留、却下のいずれか |
| 根拠レース | 判断の根拠になったレースや検証ログ |
| 次回反映メモ | 次回予想時に反映・再検証する具体メモ |

## 橋本競馬AI学習ログシステムの保存ルール

1. 競馬場別の運用ログは、必ず該当する競馬場フォルダの年別フォルダへ保存します。2026年分は `各競馬場フォルダ/2026/` を使用します。
2. 事前予想は `2026/事前予想/` へ保存します。ファイル名は `YYYY-MM-DD_競馬場名_R番号_レース名_事前予想.md` を基本形にします。
3. 結果検証は `2026/結果検証/` へ保存します。的中・不的中、回収率、ズレた仮説、次回修正点を必ず残します。
4. OS更新は `2026/OSアップデート/` へ保存します。馬場バイアス、脚質補正、枠順補正、距離別傾向など、競馬場OSへ反映する更新案を記録します。
5. 総括・学習ログは `2026/保存ログ/` へ保存します。開催日単位の総括、AI判断の成功・失敗、次回へ引き継ぐ改善メモを集約します。
6. 共通AI機能、特徴量、モデル改善案は `AI研究所/` へ保存し、競馬場固有ログと混在させません。

## 運用ルール

- 競馬場固有の傾向、開催日の馬場、コース別メモ、レース総括は、該当する競馬場別フォルダに保存します。
- 各競馬場の運用データは、`事前予想/`、`結果検証/`、`OSアップデート/`、`保存ログ/` に分け、予想前・予想後・ルール改善・総括学習を混在させないようにします。
- 複数競馬場で使える判断ルールは、`競馬場OS/` に整理してからAI研究や各DBへ反映します。
- AIモデル、特徴量、重み付け、評価指標、改善仮説は、`AI研究所/` に保存します。
- 的中・不的中にかかわらず、買い目と結果は `結果検証/` または各DBの検証領域に残します。
- 騎手と調教師の補正材料は、競馬場別フォルダではなく、それぞれ `騎手データベース/` と `調教師データベース/` に集約します。
- 高配当、危険人気馬、神レースの再現条件は、対応するDBに蓄積し、次回予想の学習材料として再利用します。

## Phase2-2 AI指数自動計算エンジン

出馬表CSV取込後、`AI指数`・`神穴指数`・`危険人気馬指数` が空欄の馬は、ブラウザ側の自動計算エンジンで即時スコアリングされます。CSVや手入力フォームで各指数が入力済みの場合は、その値を「手入力」として優先し、未入力の指数だけ「自動計算」で補完します。

### 計算関数

`src/dashboard.js` で以下の関数をグローバル公開しています。

- `calculateAiScore(horse)`
- `calculateDarkHorseScore(horse)`
- `calculateRiskScore(horse)`
- `calculateAllHorseScores(horses, raceContext)`

### AI指数

AI指数は、人気補正、オッズ補正、脚質補正、想定4角位置補正、調教評価補正、馬場状態補正、距離補正、危険人気馬減点を合算し、0〜100点に丸めます。危険人気馬指数が高い人気馬はAI指数から自動減点されます。

### 神穴指数

神穴指数は、人気妙味、中穴〜大穴補正、脚質展開一致、想定4角位置、調教評価、オッズ妙味、危険人気馬ではないことを評価します。6人気以下または高オッズでも、脚質・調教・4角位置が噛み合う馬を神穴ランキングへ押し上げます。

### 危険人気馬指数

危険人気馬指数は、1〜3人気かつ低オッズ、脚質の展開不利、想定4角位置の悪さ、調教評価の低さ、馬場状態と脚質の不一致、距離適性不明を加点して算出します。危険指数80以上の人気馬は三連単生成で1着候補から除外されます。

### CSV取込後の使い方

1. 「実データ入力」内のCSV取込へ、出馬表CSVを選択します。
2. `AI指数`・`神穴指数`・`危険人気馬指数` が空欄の行は、取込時に自動計算値が入ります。
3. 入力済みの指数は手入力値として保持されます。
4. 出走馬一覧の「計算」列で、各指数が `自動計算` か `手入力` かを確認できます。
5. 自動計算後の指数は、AI指数ランキング、神穴ランキング、危険人気馬ランキング、三連単生成、WIN5候補へ即時反映されます。
