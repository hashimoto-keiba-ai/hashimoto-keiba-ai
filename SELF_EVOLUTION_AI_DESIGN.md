# 橋本競馬AI Phase2-4 自己進化AI強化設計

## 目的

バックテスト、結果検証、OSアップデートを自己進化AIへ接続し、AI指数や各AI補正を継続的に改善できる設計を整理します。

本設計は、既存の予想画面、結果検証ログ、ROI管理、WIN5、競馬場OS、距離別OSを壊さず、改善候補を段階的に反映するためのドキュメントです。即時自動反映ではなく、ログ化、検証、採用判断、次回予想への補正メモ表示を基本方針とします。

## 1. 自己進化AIの目的

自己進化AIは、予想と結果の差分を継続的に学習し、次回以降の予想精度と資金効率を高めることを目的とします。

- **的中率改善**
  - AI指数上位馬、軸候補、相手候補の的中条件を蓄積する
  - 的中した展開、馬場、距離、枠順、脚質、騎手/調教師条件を再利用する
  - 不的中だった条件は過信しないように補正する
- **回収率改善**
  - ROIが高かった買い目構造、人気ゾーン、券種、点数配分を分析する
  - 的中しても回収率が低い条件は資金配分AIで抑制する
  - 高配当につながった条件は万馬券AI、神穴AI、WIN5 AIへ反映する
- **危険人気馬検出精度改善**
  - 危険人気馬判定が成功した条件を蓄積する
  - 危険判定した馬が好走した失敗条件を記録する
  - 人気、オッズ、距離不安、馬場不利、展開不利、ローテ、枠順、騎手乗替の重みを見直す
- **神穴検出精度改善**
  - 神穴候補が馬券内に来た条件を強く評価する
  - 神穴候補が不発だった条件を減点する
  - 人気薄でも走れる展開、馬場、距離、脚質、枠順、騎手/調教師条件を抽出する
- **WIN5構造精度改善**
  - 堅軸、準堅軸、波乱、超波乱の分類精度を改善する
  - A固定、B本線、Cゾーン、D爆発候補の点数配分を見直す
  - WIN5対象レース全体の人気ゾーン構造と荒れ方を学習する

## 2. 入力データ

自己進化AIは、以下のデータを入力として扱います。

| 入力データ | 主な内容 | 利用目的 |
| --- | --- | --- |
| 事前予想ログ | 予想時点のAI指数、印、買い目、危険人気馬、神穴候補、WIN5候補 | 予想時点の判断根拠を保存し、結果との差分比較に使う |
| 結果検証ログ | 着順、払戻、的中/不的中、反省点、予想との差分 | 成功条件と失敗条件を抽出する |
| OSアップデートログ | 競馬場別、距離別、馬場別、脚質別の更新案 | 競馬場OS・距離別OSへの反映候補を管理する |
| バックテスト結果 | 過去レースに対する的中率、回収率、期待値、条件別成績 | ルール変更の妥当性を検証する |
| ROI記録 | 投資額、払戻額、回収率、券種別/競馬場別/月別ROI | 資金配分AIと買い目構造の改善に使う |
| WIN5結果 | 対象レース、的中状況、各レース分類、人気ゾーン、点数 | WIN5構造AIの分類精度を改善する |
| 自己進化ログ | 改善対象、変更前後ルール、理由、根拠レース、採用状態 | 改善履歴を一元管理し、次回予想へ接続する |

## 3. 改善対象AI

自己進化AIは、単一の指数だけでなく、予想、買い目、資金配分、OS補正を横断して改善します。

- **AI指数**
  - 基礎能力、近走成績、コース適性、距離適性、馬場適性、騎手/調教師補正の重みを更新する
- **展開AI**
  - 逃げ、先行、差し、追込の有利不利をレース条件別に補正する
  - ペース想定のズレを検証し、展開利/展開不利の判断を更新する
- **神穴AI**
  - 人気薄が好走した条件を強加点する
  - 不発だった穴条件を減点し、過剰な穴狙いを抑制する
- **危険人気馬AI**
  - 人気馬が凡走した条件を加点する
  - 危険判定した人気馬が好走した条件を減点する
- **万馬券AI**
  - 高配当に直結した組み合わせ、人気ゾーン、荒れパターンを更新する
  - 点数過多になった条件はROIと合わせて抑制する
- **WIN5 AI**
  - 堅軸/波乱分類、A/B/C/Dゾーン配分、人気ゾーン構造を更新する
  - 1レース単位だけでなく5レース全体の点数効率を改善する
- **資金配分AI**
  - 的中率だけでなくROIを基準に購入比率を見直す
  - 低期待値の本線過多、高期待値の穴不足を補正する
- **競馬場OS**
  - 競馬場別の枠順、脚質、馬場、直線、コーナー、坂、開催時期の補正を更新する
- **距離別OS**
  - 短距離、マイル、中距離、長距離、ダート距離別の適性補正を更新する

## 4. 改善ロジック

改善ロジックは、成功条件を加点し、失敗条件を減点し、競馬場別・距離別のOSへ段階的に反映する流れで管理します。

### 基本ルール

- **的中した条件を加点**
  - AI指数上位、展開利、馬場適性、距離適性、枠順有利、騎手/調教師補正など、的中に寄与した条件を加点する
  - 複数レースで再現した条件は採用候補へ昇格する
- **外れた条件を減点**
  - 過大評価した指数要素、展開読み違い、馬場読み違い、距離適性の誤判定を減点する
  - 1回の外れだけで即時削除せず、保留ステータスで再検証する
- **危険人気馬判定成功を加点**
  - 危険人気馬が実際に凡走した場合、判定理由の条件を加点する
  - 特に上位人気かつ低着順だった場合は、危険判定の信頼度を高める
- **神穴的中を強加点**
  - 神穴候補が馬券内に入った場合、通常的中より強く加点する
  - 人気薄、展開利、馬場バイアス、枠順、脚質、騎手/調教師穴傾向を重点的に保存する
- **人気ゾーン構造を更新**
  - 1〜3番人気中心、4〜6番人気混入、7番人気以下台頭など、レースの人気構造を記録する
  - 三連単、万馬券、WIN5の買い目生成に反映する
- **競馬場別OSへ反映**
  - 競馬場ごとの枠順、脚質、馬場、距離、開催時期の傾向として更新案を作成する
  - 更新案は自己進化ログで採用/保留/却下を管理する
- **距離別OSへ反映**
  - 距離ごとの先行有利、差し有利、スピード型、スタミナ型、血統傾向を更新案として保存する
  - 競馬場OSと矛盾する場合は、根拠レース数とROIで優先度を判断する

### 反映ステータス

| status | 意味 | 次の扱い |
| --- | --- | --- |
| proposed | 改善候補として抽出済み | バックテストまたは追加検証へ回す |
| testing | バックテスト中または実戦検証中 | 次回予想では注意メモとして表示する |
| adopted | 採用済み | AI補正テーブル、競馬場OS、距離別OSへ反映する |
| pending | 保留 | 根拠レース不足として継続観察する |
| rejected | 却下 | 反映しないが、同条件の再発確認用に履歴は残す |

## 5. 自己進化ログ形式

自己進化ログは、改善の根拠と反映状態を追跡できるよう、以下の項目を標準形式とします。

| 項目 | 型/例 | 内容 |
| --- | --- | --- |
| date | `2026-06-02` | 改善ログ作成日 |
| racecourse | `東京` | 対象競馬場 |
| distance | `芝1600m` | 対象距離・条件 |
| target_ai | `神穴AI` | 改善対象AI |
| before_rule | `差し穴の加点 +3` | 変更前ルールまたは変更前補正 |
| after_rule | `差し穴の加点 +5` | 変更後ルールまたは変更後補正案 |
| reason | `稍重の外差し馬場で人気薄差し馬が連続好走` | 変更理由 |
| evidence_race | `2026-05-31 東京11R` | 根拠レース |
| status | `proposed` | proposed/testing/adopted/pending/rejected のいずれか |
| next_usage_note | `次回の東京芝1600mで外差し神穴候補を強調表示` | 次回予想画面に表示する改善メモ |

### JSON例

```json
{
  "date": "2026-06-02",
  "racecourse": "東京",
  "distance": "芝1600m",
  "target_ai": "神穴AI",
  "before_rule": "差し穴の加点 +3",
  "after_rule": "差し穴の加点 +5",
  "reason": "稍重の外差し馬場で人気薄差し馬が連続好走したため",
  "evidence_race": "2026-05-31 東京11R",
  "status": "proposed",
  "next_usage_note": "次回の東京芝1600mで外差し神穴候補を強調表示する"
}
```

## 6. Phase2-4 実装順

### Step1 自己進化ログの標準化

- 自己進化ログの項目を `date`、`racecourse`、`distance`、`target_ai`、`before_rule`、`after_rule`、`reason`、`evidence_race`、`status`、`next_usage_note` に統一する
- 既存ログがある場合は不足項目を空欄または暫定値で補完する
- 採用/保留/却下/検証中のステータスを明確にする

### Step2 バックテスト結果との連動

- バックテスト結果から的中率、回収率、期待値、危険人気馬判定成功率、神穴馬券内率を集計する
- 条件別の成功/失敗を自己進化ログへ自動または手動で登録する
- バックテストで再現しない改善案は `pending` または `rejected` として扱う

### Step3 OSアップデートとの連動

- 自己進化ログの `racecourse` と `distance` を使って、競馬場OS・距離別OSの更新候補を作る
- OSアップデートログには、根拠レース、変更理由、採用状態を残す
- 競馬場OSと距離別OSで補正が競合する場合は、根拠レース数、直近成績、ROIを優先判断材料にする

### Step4 AI指数補正テーブル化

- AI指数、展開AI、神穴AI、危険人気馬AI、万馬券AI、WIN5 AI、資金配分AIの補正値をテーブル化する
- 補正テーブルには対象条件、補正値、根拠、適用期間、ステータスを持たせる
- 採用済み補正のみ通常計算へ反映し、検証中補正は画面上の参考メモとして扱う

### Step5 競馬場OSへ反映

- 採用済みの競馬場別傾向を競馬場OSへ反映する
- 競馬場OSには、枠順、脚質、馬場、距離、開催時期、人気ゾーンの補正を保存する
- 反映後も自己進化ログに履歴を残し、後から戻せるようにする

### Step6 次回予想画面へ改善メモ表示

- 次回予想画面に `next_usage_note` を表示する
- 競馬場、距離、対象AIが一致する改善メモを優先表示する
- `testing` と `adopted` は強調表示し、`pending` は参考情報として表示する
- 危険人気馬、神穴、WIN5、資金配分に関わるメモは買い目生成前に確認できるようにする

## 運用ルール

- 1レースだけの結果で強制採用しない
- 神穴的中や危険人気馬判定成功は強く評価するが、バックテストで再現性を確認する
- ROIが悪化する補正は、的中率が改善しても採用を保留する
- 競馬場OS・距離別OSへの反映は、自己進化ログの履歴を残してから行う
- 既存機能を壊さないため、最初はドキュメント、ログ形式、補正テーブル案から段階的に実装する
